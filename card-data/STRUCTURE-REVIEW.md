# Revelio – Data Structure Review

After adding a second language (German Base Set), here's a review of the JSON
structure with concrete recommendations.

## What works well

- Clean split: language-independent gameplay at top level, printed face per language
  under `localizations`.
- `status`/`source` per localization makes the hybrid (official/machine) model explicit.
- Stable `id` slugs, type-specific blocks nullable, validates cleanly.

## Issues & recommendations

### 1. Redundant text on Adventure/Match cards (fix recommended)
For Adventure cards the same wording sits in **both** `text` and `adventure.effect`
(per language). This duplicates content and risks drift between languages.
→ **Pick one canonical form.** Keep the structured `adventure{effect,toSolve,reward}`
/ `match{toWin,prize}` and set `text` to `null` for those types (the UI renders text
from the structured fields). One source of truth per field.

### 2. Set metadata repeated on every card (normalize)
`setName`, `releaseDate`, and `isOfficial` are copied onto all 1,035 cards.
→ **Move set info into a `sets.json`** keyed by `setCode` (name, releaseDate,
isOfficial, total cards, set symbol image). Cards keep only `setCode` + `number`.
Removes repetition and gives one place for set symbols/metadata.

### 3. Source of truth vs build artifact (process)
`cards.json` is now ~1 MB with 2 partial languages; it will grow per language and
produces noisy git diffs.
→ **Treat `cards.json` as a build output.** Editable sources stay small and
git-friendly: per-set English (from hpjson) + per-language overlays
(`translations/<lang>.json`, already in place). A build step assembles the bundle(s).

### 4. Client payload / search (plan)
A search UI shouldn't load every language for every card, nor scan raw JSON.
→ Build **derived artifacts**: (a) a slim per-language file or a single bundle, and
(b) a **search index** with diacritics-folded, lowercased name+text+flavor per
language. Keep these out of the source tree (generated).

### 5. Derivable convenience fields (document, keep)
`languages[]` duplicates `Object.keys(localizations)`, and top-level `name` duplicates
`localizations.en.name`. Both are handy for consumers/sorting.
→ Keep them, but document that they are **derived** and always regenerated by the
build (never hand-edited), so they can't drift.

## Suggested target layout

```
card-data/
  sets.json                 # set metadata keyed by setCode  (new)
  card.schema.json          # + sets.schema.json             (new)
  sources/                  # editable English per set (from hpjson)  (optional split)
  translations/<lang>.json  # editable overlays per language
  build_dataset.py                  # transform + merge + index  (one entry point)
  dist/                     # generated, git-ignored
    cards.json              # full bundle, all languages
    cards.<lang>.json       # slim per-language
    search-index.<lang>.json
```

## Priority

1. De-duplicate Adventure/Match vs `text` (data correctness).
2. Normalize `sets.json` (removes most repetition).
3. Single `build_dataset.py` entry point (+ `dist/`), mark derived fields.
4. Search index when we build the prototype.
